## 1. 状況整理

- ビーンズ歯科様の本番は `main` ブランチ (`346bba3`, 2026-03-17) で稼働中
- `dev` ブランチには **予約周りの修正・機能追加が 12 コミット先行**しており、本番にはまだ適用されていない
- 報告された症状: 「イージーアポの予約ができていない」

`dev` で行われている修正の多くは、本番運用で起こりがちな患者特定・電話番号衝突・営業時間制限などの **実運用バグ修正**である可能性が高い。優先度順に原因仮説を整理する。

## 2. 原因仮説（main 適用済み / dev 適用済みの差分から）

### 仮説A（最有力）: 電話番号検索の精度不足 — `81f65de` 未適用
- main の電話番号検索は dev で全面書き換えられている（[src/pages/AppointPage.ts](../src/pages/AppointPage.ts) に +89/-18 行）
- dev では次の挙動に変更：
  1. **下4桁で API 検索 → 結果を tel1 の末尾一致で絞り込む**
  2. tel1 で見つからなければ患者詳細を再取得して **tel2 も評価**
  3. 比較時、短い方が最低6桁を満たさないと不一致扱い
- 旧実装ではフル番号の完全一致など精度ガタの可能性。**家族で同一電話番号を共有するクリニックでは別人ヒット → 別人IDで予約 → 失敗 or 重複** が起きやすい

### 仮説B: 患者特定の精度不足 — `7ba0705` + `e7c4fa6` 未適用
- dev で `[patient]` タグによる患者特定経路が大幅改善
- `7ba0705`: 患者番号 + 生年月日での特定を電話番号検索より **優先化**
- `e7c4fa6`: 検索エンドポイントを `/patients/number/` に切替（電話番号が異なっていても患者特定可能）
- これらが main に未適用 → 既存患者が捕捉できず、**新規患者扱いで予約しようとして EasyApo 側でバリデーション失敗**するパターンがありうる

### 仮説C: フリガナ名で登録できていない — `692bc55` 未適用
- 患者特定後に **入力名ではなく `name_kana`** を予約の患者名に上書きする修正
- main では入力された口頭名（漢字推測やカタカナ揺れ）でそのまま予約が作られる
- 結果として「予約APIは成功するが、患者マスタと別人として登録 → 後の照合や受付確認で失敗扱い」になる可能性

### 仮説D: websetting 制限未適用 — `6743735` 未適用
- dev では `/websetting` API から **予約可能期間・曜日別受付時間・締切時間**を取得し、空き枠取得・予約作成の両方に適用
- main にはこの制限ロジックがない → 営業時間外/締切後の時間枠を案内 → 予約API で弾かれる可能性
- ※ ビーンズ歯科様の現プロンプトには昼休み（12:30-13:00, 17:30-18:00）が示唆されており、これらが websetting に乗っている可能性がある

### 仮説E: メニュー（external_menu_id）の不整合
- プロンプトでは `13650`〜`24621` 系の external_menu_id が指定されている
- これらが **現行 EasyApo マスタに存在するか** は本リポジトリでは判定できない。実運用ログで「menu not found」相当のエラーが出ていないか要確認
- ビーンズ歯科様の現プロンプトの `予約メニュー選択ルール` が EasyApo マスタとずれていると、`create_reservation` 段階で必ず失敗する

### 仮説F: 担当者衝突時のフォロー欠落 — `f87e85c` 未適用
- 予約変更で担当者が時間にいない場合、main は何もケアしない
- dev は `pic` コンテキストに担当者を保持し空き枠を絞り込む
- 「予約変更ができない」症状ならこれが該当

### 仮説G: 再ログイン時キャッシュ汚染 — `17275c5` 未適用
- main は再ログイン時に `treatmentItems` キャッシュをクリアしない
- セッション切れ後に古いメニュー情報を参照 → `duration_min` がズレ／メニュー存在判定が誤判定する可能性

### 仮説H: resource_ids/notes 未対応 — `280cd2b` 未適用
- 呼び出し元（SmartCall 本体）が `resource_ids` や `notes`（`[patient]` タグ等）を送っていたら、main は読み取れず無視している可能性
- 呼び出し元の仕様変更とずれていれば「予約は通るが内容が違う」「`[patient]` タグでの患者検証ができない」状況になる
