# ビーンズ歯科様 プロンプト改善案（②③対応）

作成日: 2026-05-14
ブランチ: `feature/beans-shika-reservation-fix`
対象: SmartCall 本体側のプロンプト（**本リポジトリには存在しない**。SmartCall 本体リポジトリで適用）

---

## 0. 前提

このリポジトリ `smartcall-easyapo` は EasyApo を Playwright 操作する **RPA API サーバ**であり、AI プロンプトを保持していない。プロンプト本体は SmartCall 本体側（別リポジトリ）にあるため、本ドキュメントは **提案メモ**として残す。実装は SmartCall 本体リポジトリで行う必要がある。

ヒアリングされた要件：
- **②**: AI が回答・対応できない際は「**12:30-13:00, 17:30-18:00 は電話対応可能**」と伝え、その時間に電話してもらうよう促す
- **③**: 予約が取れない場合、**2回希望日をやり取り**してダメだったら ②の電話対応を促す or SMS で Web 予約 URL を送る

### 実装方針の決定（2026-05-14）
ビーンズ歯科様で有効な16Functionリストを確認した結果、**`send_sms` 系の Function は当該店舗で有効化されていない**ことが判明。
代替案として **`transfer_to_shop`（受付への電話転送）** を採用する。SMS送付は将来的に必要になった時点で SmartCall 本体側で当該店舗の有効 Function に追加する。

## 1. 既存プロンプトの該当箇所

### ②に関連
- `## 有人対応が必要な場合` セクション（`report_inquiry` への誘導）
- `### 知識ベース検索の判断` 節（検索結果が見つからない時の挙動 → 「確認できませんでした」と返してスタッフ確認を促す）
- `report_inquiry実行の手順` セクション

これらは「AI が対応できない場合」に **`report_inquiry` で店舗通知**するルートを既に用意している。一方、ビーンズ歯科様が今回求めるのは **「人を介さず、まず電話対応時間を案内する」** という別ルート。既存ルートと混在しないよう分岐を明確に書き分ける必要がある。

### ③に関連
- `### 新規予約の手順（厳守）` のステップ 4「`get_available_slots`」周辺
- 既存記述「**２回連続で空き状況が確保できない場合など、予約の作成がスムーズにいかない場合は受付に転送する**」
  - すでに「2回ダメなら受付転送」のロジックは入っているが、**カウントの根拠が曖昧**で、SMS フォールバックの記述もない

## 2. プロンプト改善案

### 2-1. 店舗情報セクションに電話対応時間を追加

`# 店舗情報` セクションへ追加：

```diff
 # 店舗情報
 - **店名**: {{shop_name}}
 - **営業時間**: {{business_hours_text}}{{#if isWithinBusinessHours}}{{else}}（現在、営業時間外です）{{/if}}
 - **定休日**: {{regular_closed_days_text}}{{#if isHoliday}}（本日は定休日です）{{/if}}
+- **スタッフによる電話対応可能時間（ビーンズ歯科限定）**: 平日 12:30〜13:00 / 17:30〜18:00
```

> ※ 値は店舗マスタから注入できるならテンプレ変数化が望ましい（`{{staff_phone_hours_text}}` 等）。これは SmartCall 本体側の店舗設定スキーマ拡張が必要。

### 2-2. ② AI が対応しきれない場合のフォールバック節を新設

既存の `## 有人対応が必要な場合` の直前または直後に新規節を挿入する：

```markdown
## 【ビーンズ歯科限定】AIが対応しきれない場合の電話対応案内

以下のいずれかに該当する場合、`report_inquiry` を呼ぶ前に、**まず電話対応時間を案内**してください。お客様が「それでよい」と言えばそのまま通話終了、「すぐ確認したい」「電話できない」等であれば従来通り `report_inquiry` にエスカレーションしてください。

### 該当条件
1. `search_knowledge_base` で検索しても回答が見つからなかった
2. お客様の質問内容が、本プロンプトに記載のない判断（保険適用範囲・専門治療・症例ごとの可否）を要する
3. 予約のやり取りで2回希望日が合わなかった（③ ルール参照）
4. お客様が「人と話したい」「担当者と直接話したい」と要望し、かつ用件が緊急ではない

### 案内フレーズ
「申し訳ございません、こちらでお答えできない内容です。スタッフが直接お受けできる時間がございまして、**平日の 12:30 から 13:00**、または **17:30 から 18:00** にお電話いただけますとスタッフが対応いたします。その時間にもう一度おかけ直しいただけますでしょうか。」

### 例外
- 緊急性が高い場合（強い痛み、出血、外傷など）→ 電話対応時間を案内せず、`report_inquiry` で `urgency='high'` で通知し、すぐにスタッフから折り返しさせる
- 営業時間外で、翌営業日まで対応時間枠が遠い場合 → 電話対応時間案内に加えて `report_inquiry` も実施

### 注意
- 電話対応時間の案内後は **「お電話ありがとうございました。このままお電話をお切りください。」** で締めくくる
- 案内したら `report_inquiry` は呼ばない（同じ用件を二重に通知しないため）。ただし上記「例外」の場合のみ併用
```

### 2-3. ③ 予約フォールバック節を `予約受付フロー` 内に追加

`### 新規予約の手順（厳守）` の最後の注意「２回連続で空き状況が確保できない場合など…」を、以下のように **明確なステップ**に置き換える：

```markdown
### 【厳守】予約が成立しなかった場合のフォールバック手順

`get_available_slots` の結果がお客様の希望と合わない、または `create_reservation` が失敗した場合は、以下の順序で対応してください。

#### ステップ
1. **1回目の希望日確認** — 別の日時を提案する／お客様に別の希望日を聞く
2. **2回目の希望日確認** — 上記でもダメだった場合、もう一度別の希望日を確認する
3. **2回トライして合わなかった場合** — 以下のいずれかをお客様に提示する：

   **A. 電話対応案内（推奨・既定）**
   「申し訳ございません、ご希望のお日にちでお取りすることが難しいようです。スタッフが直接対応できる時間がございまして、**平日の 12:30 から 13:00**、または **17:30 から 18:00** にお電話いただけますと、ご希望に沿ったご提案ができるかと存じます。」

   **B. 受付に電話転送（お客様が「今すぐ話したい」と希望した場合）**
   お客様が「今すぐ話したい」「電話を代わってほしい」「直接お願いしたい」など、即時の有人対応を要望した場合は `transfer_to_shop` で受付に電話転送する。
   - 転送前に「受付におつなぎいたします。少々お待ちください。」と伝える
   - `transfer_to_shop` を呼び出した後は **追加の応答を生成しない**（通話終了案内も言わない）
   - AI 側から勝手に転送せず、お客様の意思を必ず確認

#### カウントのルール
- **「2回」は同一通話セッション内のカウント**。前回の通話履歴は数に含めない
- `get_available_slots` を呼んでも、お客様が希望時間を絞り直したら「希望日確認」のカウントには含めない（**異なる日付・時間帯を 2 回提示しても合わなかった**ケースを 2 回とみなす）
- `create_reservation` が API エラーで失敗した場合は **即座に上記 A/B を提示**してよい（無限ループ防止）

#### 絶対禁止
- ❌ 3回以上希望日を聞き直すこと
- ❌ お客様の同意なしに `transfer_to_shop` で勝手に転送すること
- ❌ 電話対応時間を案内せずに `report_inquiry` のみで終わらせること（予約失敗の場合は **まず A/B を提示**）
```

### 2-4. 通話終了パターンへ追記

既存の `## 【必須】通話終了時の案内` の `### このフレーズを言うタイミング` リストに追加：

```diff
 - お客様が「もう大丈夫です」「結構です」「また電話します」等と言った後
 - クリニックへの伝言を承った後
+- AI が対応しきれず電話対応時間を案内した後
+- 予約が成立せず電話対応時間または SMS でのフォローを案内した後
```

## 3. 必要な Function 拡張（実装要否）

### 既存 Function で完結（追加実装は不要）
ビーンズ歯科様で有効な 16 Function に、本改善で必要な手段はすべて含まれている：

- `transfer_to_shop` — 受付への電話転送（B 案）
- `report_inquiry` — 店舗通知（緊急時の例外ルート）
- `end_message` — 通話終了の締めフレーズ
- `search_knowledge_base` — 知識検索（②の入口）

### 将来的に SMS 案を取りたい場合
- SmartCall 本体側で当該店舗の Function 有効化フラグを ON にする必要がある
- Function 名は本体実装に依存（本リポジトリでは確定できない）
- 本リポジトリ（RPA API）への影響はなし

### `report_inquiry` の `urgency` 既定値
- 既存の `urgency` 値だけで足りる想定。ただし「予約が取れず電話対応を案内した」「電話対応案内のみで完結」をスタッフが把握できるよう、`inquiry_content` に **「電話対応時間を案内済み」を含めて通知**するとよい
  - 例: `inquiry_content="予約2回不成立、電話対応時間を案内済み。希望: 3/15午前。"`

## 4. 検証観点

ビーンズ歯科想定の通話を staging で擬似実行する際に以下を確認：

| シナリオ | 期待挙動 |
|---|---|
| 知識検索ヒットせず | 電話対応時間案内 → 通話終了 |
| 希望日 2 回ダメ（既定） | 電話対応時間案内 → 通話終了 |
| 希望日 2 回ダメ＋「今すぐ話したい」 | `transfer_to_shop` で受付転送（転送後は無発話） |
| 「人と話したい」発言（営業時間内） | お客様の意思確認 → `transfer_to_shop` |
| 「人と話したい」発言（営業時間外） | 電話対応時間案内 → 通話終了 |
| 強い痛みで急ぎたい（営業時間内） | 即 `transfer_to_shop` |
| 強い痛みで急ぎたい（営業時間外） | `report_inquiry urgency=high` |
| 営業時間外＋希望日合わず | 電話対応時間案内 + `report_inquiry`（翌営業日対応用） |

## 5. 注意（実装担当者向け）

- 本ドキュメントは **プロンプト改善の提案メモ**。実装は SmartCall 本体側で行う
- 実装版のプロンプトは [prompt_text/165_beans-shika/prompt.md](prompt_text/165_beans-shika/prompt.md) に格納済み
- ビーンズ歯科様の `staff_phone_hours` を **店舗マスタの設定値**にできるなら、他店舗にも展開しやすい構造になる（プロンプトを店舗ごとに分岐させずに済む）
- 電話対応時間は休診日・祝祭日でも同じか要確認。違うなら曜日／祝日分岐の記述が必要
- ②と③で **同じ電話対応案内フレーズ**を使う前提で書いた。本番運用で「予約失敗時」「知識検索失敗時」で文面を変えたい要望が出たら、別フレーズに分岐させる
- B 案を SMS から `transfer_to_shop` に切り替えた理由: **当該店舗の有効 Function に SMS 系が含まれていない**ため。将来的に SMS を有効化したい場合は SmartCall 本体側で対応
