# Ci EasyApo 2 API 仕様（Playwright実機採取）

採取日: 2026-06-17
採取元: `https://cieasyapo2.ci-medical.com/`（57小西歯科様 本番アカウント）
採取方法: PlaywrightでUI操作 + ブラウザ内 `fetch()` 直叩きで実APIをcapture

---

## 1. なぜこの仕様書が必要か

Ci EasyApo 2 のフロントエンドが **Vue 2 → Vue 3 (v3.5.32)** に移行されたことにより、`smartcall-easyapo` の既存実装（`el.__vue__.$vnode?.tag?.endsWith(...)` や `reserveDay.$store.commit(...)` で Vue インスタンスに直接アクセス）が **すべて動作しなくなった**。

採取の結果、UI（DOM）は **API のレスポンスをVueが描画している薄いレイヤー**であり、サーバAPIの契約は維持されていることが判明した。したがって、Vue インスタンス経由をやめて **ブラウザ内 fetch() で API を直叩きする方針（Browser-to-API）** に書き換える方が、UI 変更耐性・Vueバージョン非依存・コードシンプル化のすべての点で有利。

本書は AppointPage 書き換えのリファレンスとなる。

---

## 2. 共通仕様

### 2-1. 認証

- ログイン: `POST /login`
- 初期化時に `GET /sanctum/csrf-cookie` を呼ぶ必要あり（Laravel Sanctum）
- セッション維持は **Cookie**（自動で送信される）
- 書き込み系（POST）には **`X-XSRF-TOKEN` ヘッダ必須**
  - 値は Cookie の `XSRF-TOKEN` を `decodeURIComponent()` した結果

```js
const xsrf = decodeURIComponent(
  document.cookie.split('; ').find(c => c.startsWith('XSRF-TOKEN='))?.split('=')[1] || ''
);
```

### 2-2. 共通ヘッダ

```
content-type: application/json
accept: application/json, text/plain, */*
x-xsrf-token: <CSRF token>            // 書き込み系のみ必須
```

### 2-3. 共通レスポンス形式

```json
{
  "result": true | false,
  "data": <object|null>,
  "message": null | { field: [string, ...] } | string,
  "error": string  // result=false 時のみ
}
```

### 2-4. キャッシュバスター

リクエスト URL の末尾に `&t=<timestamp>` を付けるのが慣習。`Date.now()` で生成。

## 5. 書き込み系 POST

### 5-1. `POST /reservations` — 予約作成

**Headers:**
```
content-type: application/json
x-xsrf-token: <token>
```

**Body:**
```json
{
  "patient_number": "" | "116092",        // 空=新規患者扱い、値=既存患者
  "reservation_date": "2026-09-29",
  "column_no": 2,
  "time_from": "09:30",
  "time_to": "09:45",
  "patient_name": "山田太郎",
  "treatment_id": null | 91545,
  "color": "#FFFFFF",
  "pic": [null, null, null],               // 担当者3スロット
  "memo": [],                              // または [{id, memo}, ...]
  "t": 1781658803995
}
```

**成功レスポンス（200）:**
```json
{ "result": true, "data": null, "message": null }
```

**注意:** レスポンスに作成された予約IDが返らない。
作成後は `GET /reservations?from=<date>&days=1` を再取得し、`patient_name + time_from + column_no` で新規予約を特定する必要あり。

### 5-2. `POST /reservations/{id}` — 予約更新

**Body:**
```json
{
  "id": 41506041,
  "column_no": 2,
  "status": 1,
  "reservation_date": "2026-10-02",         // 変更後の日付
  "time_from": "09:30",
  "time_to": "09:45",
  "treatment_id": null,
  "patient_name": "テスト 動作確認",
  "color": "#FFFFFF",
  "pic": [null, null, null],
  "memo": [],
  "patient_number": "",
  "t": 1781659610379
}
```

**注意:** 既存予約の全フィールドを送る形式（部分更新ではない）。`GET /reservations/{id}?original=true` で取得した値をベースに、変更フィールドだけ書き換えて送るのが安全。

**成功レスポンス（200）:**
```json
{ "result": true, "data": null, "message": null }
```

**バリデーションエラー例（422）:**
```json
{
  "result": false, "data": null,
  "message": { "patient_number": ["指定された 患者ID は見つかりませんでした"] },
  "error": "指定された 患者ID は見つかりませんでした"
}
```

### 5-3. `POST /reservations/{id}/cancel` — 予約キャンセル/削除

**Body:**
```json
{
  "circumstance_type": 1 | 99,         // 1=キャンセル（履歴残す）, 99=削除（履歴も消す）
  "cancel_type": 1,                    // 必須！ 1=電話キャンセル等。99の場合も必須
  "t": 1781659712345
}
```

**⚠️ 重要：旧コードでは `circumstance_type=99` 時に `cancel_type` を undefined にしていたが、サーバ仕様は必須。書き換え時に修正必要。**

**成功レスポンス（200）:**
```json
{ "result": true, "data": null, "message": null }
```

**バリデーションエラー例（422）:**
```json
{
  "result": false, "data": null,
  "message": { "cancel_type": ["種類 を入力してください"] },
  "error": "種類 を入力してください"
}
```
