## ローカル運用スクリプト群

いずれも **コンテナ名 `smartcall-easyapo` を直接 grep する** ため、`docker-compose.yml` 前提。本番系（per-shop コンテナ）には使えない。

### `start.sh`
- `.env` 存在確認 → 既存コンテナがあれば `docker compose down` → `docker compose build && up -d`
- 起動後にポート（**3000 と表示しているがホスト側は 3001**。表示はバグ気味なので注意）、ヘルスチェックURL、停止/ログコマンドを案内
- 最も典型的なローカル起動エントリポイント

### `stop.sh`
- `docker ps` でコンテナ存在を確認し、あれば `docker compose down`
- 起動していなければ「コンテナは起動していません」と通知して終了

### `restart.sh`
- `stop.sh + start.sh` 相当を内部で順次実行
- コードを書き換えてイメージを作り直したいときに使う

### `status.sh`
- コンテナ存在チェック
- `http://localhost:3000/health` を `curl -sf`（**ホスト側は 3001 マップだが内部から見ると 3000**。スクリプトはローカル開発前提で書かれており、コンテナを通さず docker net で見るならOK、ホストから叩くなら 3001 に直す必要あり）
- `/session/status` も併せて取得して表示

### `logs.sh`
- `-f` / `--follow` でリアルタイム表示
- `--tail N` で末尾 N 行（デフォルト 100）
- 起動していなければ起動方法を案内して終了

## デプロイスクリプト群（共通フロー）

すべて同一構造（[deploy-smartcall-easyapo-beans-shika.sh:42-174](../scripts/deploy-smartcall-easyapo-beans-shika.sh#L42-L174) 参照）：

1. **ローカルリポジトリ確認** — 未コミット変更があれば警告し続行可否を確認
2. **GitHub 同期確認** — `origin/main` との差分を比較。`SKIP_GIT_SYNC=1` でスキップ可
3. **tarball 化** — `node_modules` / `.git` / `dist` / `screenshots` / `input` / `output` を除外
4. **踏み台へ scp** — `centos@153.126.214.207` 経由
5. **RPA サーバへ転送・展開** — 二段 SSH で対象サーバへ scp、リモートで `tar -xzf`
6. **Docker build & up** — `sudo docker compose -f <compose> build && up -d`
7. **ヘルスチェック** — `curl localhost:<port>/health` を叩いて結果表示
8. **クリーンアップ** — ローカル / 踏み台 / RPA サーバの tarball を削除

### 共通の前提条件
- SSH 鍵 `~/.ssh/milestone` を保有していること（踏み台 / RPA サーバ共通）
- ターゲット RPA サーバへの SSH/scp は踏み台経由の二段（インターネットから直接到達できない）

### 環境変数
| 変数 | 内容 |
|---|---|
| `SKIP_GIT_SYNC=1` | GitHub との同期確認をスキップ |

- 旧名 `deploy-smartcall-easyapo.sh` から `81e47f7` でリネーム → 後に `30f07f3` 相当で削除されたとされていたが、現状 main にリネーム後ファイルとして存在（残存）。実運用ステータスは要確認

### `deploy-smartcall-easyapo-staging.sh` 固有
- `--exclude='.env'` で tarball から `.env` を除外（`30f07f3`）
- ヘルスチェックは単一ポート `3011`

### `deploy-smartcall-easyapo-beans-shika.sh` / `yuki-dental.sh` 固有
- コメントに「RPA04 (`192.168.20.73`)」「ステージングサーバ経由でアクセス」と明示
- 初回用に `mkdir -p $RPA_REMOTE_PATH` を踏み台越しに実行する一手間が入っている
- 完了時にアクセスURLとログ確認コマンドを案内

## 注意点

- 運用スクリプト（start/stop/restart/status/logs）はローカル用に固定。本番系の per-shop コンテナを操作したい場合は対象サーバ上で `docker compose -f docker-compose.<env>.yml ...` または `docker logs <container_name>` を直接叩く
- デプロイスクリプトは **本番イメージを毎回ビルド**する（差分転送ではなく tarball 全置換）。リリース時間は build 時間に依存
- prod 系で `.env` が tarball に含まれる問題は staging スクリプトの除外設定を他にも展開するのが安全
- 旧 RPA01（`192.168.20.70`）のコンテナが残っている可能性あり。停止・整理計画は引き継ぎドキュメント参照
